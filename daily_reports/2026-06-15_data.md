## Findings

### Logic Errors

#### delete_loader.cc:92-170, 203-241 — Resource leak: Reader not closed on error paths
- **Severity:** medium
- **Description:** Both `LoadPositionDelete()` and `LoadEqualityDeletes()` call `reader->Close()` only after their while-loops complete successfully (lines 170 and 237). If any operation inside the loop fails (e.g., `Next()`, `Insert()`, `Reset()` returning an error), the function returns early via `ICEBERG_RETURN_UNEXPECTED` without closing the reader. The reader may hold open file handles, causing resource leaks.
- **Fix:** Wrap the reader in an RAII guard or use a scope guard that calls `reader->Close()` on scope exit regardless of success/failure. Example: `internal::ArrowArrayStreamGuard reader_guard` or a custom deleter that calls Close.

#### delete_filter.cc:700-721 — Thread-safety documentation mismatch with mutex use
- **Severity:** low
- **Description:** The `EqDeletedRowFilter()` method holds `eq_mutex_` during cache initialization (line 707) but the returned lambda captures a raw `this` pointer (line 709). The comment at line 165 says "The returned predicate is NOT thread-safe." However, the double-checked locking pattern (ensuring load + lock, then assign) is only partially correct — there's no release/acquire barrier on `eq_deleted_row_filter_cache_` itself. If `eq_mutex_` is the only synchronization, then the load at line 708 (before the lock) is unsynchronized and could observe a partially-written `std::function`.
- **Fix:** Either move the mutex lock before the load check (traditional double-checked lock), or mark `eq_deleted_row_filter_cache_` as `std::atomic` with appropriate memory ordering, or simply always lock before checking as the overhead is negligible for this path.

#### position_delete_writer.cc:63-66 — Hard failure when Write() called with buffered deletes
- **Severity:** low
- **Description:** `Write(ArrowArray* data)` at line 63 uses `ICEBERG_PRECHECK(buffered_paths_.empty(), ...)` which crashes/hard-fails if there are pending buffered position deletes. A caller who first calls `WriteDelete()` then accidentally calls `Write()` (for batch data) gets an unrecoverable error. Java's position delete writers don't support mixing batch and single writes in this way, but a more graceful approach would be to auto-flush the buffer before accepting the batch write, or return a recoverable Status error.
- **Fix:** Either auto-flush the buffer before accepting the batch write, or return an `InvalidArgument` status instead of a PRECHECK crash so the caller can recover.

#### delete_filter.cc:755-789 — Integer truncation of alive row indices
- **Severity:** medium
- **Description:** `ComputeAliveRows()` at line 749-751 finds that `batch.length <= int32_t max`, then at line 785 casts the loop index `i` (`int64_t`) to `int32_t` for `result.indices`. This is correct per the precondition check. However, if the batch length is between `INT32_MAX + 1` and the maximum possible ArrowArray length (~2^63-1), the precondition would be violated and the function would fail. While Arrow array lengths this large are practically impossible, the check uses a static precondition rather than `ICEBERG_RETURN_UNEXPECTED`, meaning the caller gets a hard crash instead of a recoverable error.
- **Fix:** Replace `ICEBERG_PRECHECK` with `ICEBERG_RETURN_UNEXPECTED` + `InvalidArgument` to allow graceful error handling.

### Java Inconsistencies

#### delete_filter.cc vs DeleteFilter.java — No "_deleted" column mode support
- **Severity:** medium
- **C++ behavior:** `DeleteFilter::ComputeAliveRows()` always returns filtered indices (rows are removed from output). There is no equivalent to Java's `hasIsDeletedColumn` / `markRowDeleted` / `_deleted` column workflow.
- **Java behavior:** Java's `DeleteFilter` supports two modes: `Deletes.filterDeleted()` (remove rows) and `Deletes.markDeleted()` (keep all rows but mark with `_deleted` boolean column), controlled by `hasIsDeletedColumn`. Java also has a `columnIsDeletedPosition()` accessor.
- **Impact:** Callers that need the "mark deleted" mode (e.g., certain engine integrations that want to see all rows with a deleted flag) cannot use the C++ implementation. Acknowledged as TODO in `file_scan_task_reader.h:43`.
- **Fix:** Implement the TODO. Add `_deleted` column support to `ComputeRequiredSchema()`, `ComputeAliveRows()`, and expose a mode flag similar to `need_row_pos_col`.

#### delete_filter.cc:214-217 vs BaseDeleteLoader.java — DeleteLoader has no caching or parallel loading
- **Severity:** high
- **C++ behavior:** `DeleteLoader` is constructed with a `FileIO` reference and loads all delete files serially in a single thread. There is no caching mechanism — each call to `LoadPositionDeletes()` or `LoadEqualityDeletes()` re-reads files from disk.
- **Java behavior:** `BaseDeleteLoader` supports: (1) overridable `canCache(long)` and `getOrLoad()` hooks for caching delete file contents, (2) a thread pool (`ExecutorService`) for parallel loading of multiple delete files via `Tasks.foreach()`, (3) size estimation to decide caching eligibility.
- **Impact:** Performance will be significantly worse in merge-on-read scans where the same position delete files are referenced by multiple data files in the same task. Java caches these; C++ re-reads them. Acknowledged as TODO in `delete_filter.h:214`.
- **Fix:** Expose a virtual factory hook (`newDeleteLoader()` or `std::function`) as noted in the TODO, and implement a caching `DeleteLoader` subclass.

#### delete_loader.cc:173-175 vs BaseDeleteLoader.java:186 — Deletion vectors not supported
- **Severity:** high
- **C++ behavior:** `LoadDV()` returns `NotSupported("Loading deletion vectors is not yet supported")`.
- **Java behavior:** `readDV()` fully supports deletion vectors — it reads the DV bytes from the file offset, deserializes using `PositionDeleteIndex.deserialize()`, and integrates them into the position delete index.
- **Impact:** Tables using deletion vectors (a V3 spec feature increasingly used for efficient position deletes) cannot be read in C++ when those DVs are present.
- **Fix:** Implement DV reading. The C++ codebase already has `PositionDeleteIndex` infrastructure; the missing piece is reading the raw bytes at the file offset and deserializing them.

#### position_delete_writer.cc:108-120 vs PositionDeleteWriter.java:123-130 — Metrics inconsistency for column_sizes
- **Severity:** low
- **C++ behavior:** When filtering delete metadata column metrics, the C++ code erases `value_counts`, `null_value_counts`, and `nan_value_counts` for the file_path and pos columns, plus conditional erasure of `lower_bounds` and `upper_bounds` when referencing multiple files. However, `column_sizes` entries for these metadata columns are **not** erased.
- **Java behavior:** `MetricsUtil.copyWithoutFieldCounts()` and `copyWithoutFieldCountsAndBounds()` create copies excluding the specified field IDs from all metrics maps including column sizes.
- **Impact:** Manifest entries for position delete files may contain column size entries for the internal `file_path` and `pos` metadata columns. While this doesn't affect correctness, it adds unnecessary data to the manifest and is inconsistent with Java behavior.
- **Fix:** Add `metrics.column_sizes.erase(path_id)` and `metrics.column_sizes.erase(pos_id)` alongside the other metric erasures.

#### delete_filter.h/cc:94-134 vs DeleteFilter.java:161-163,106-118 — No injectable DeleteLoader factory
- **Severity:** medium
- **C++ behavior:** `DeleteFilter` constructs a concrete `DeleteLoader` directly from `FileIO` in the constructor (line 520: `delete_loader_(std::move(io))`). There is no way to replace or customize the loader — no factory function, no virtual method, no template parameter.
- **Java behavior:** `DeleteFilter.newDeleteLoader()` is a protected method that subclasses can override. For example, Spark's `SparkDeleteFilter` overrides it to provide a caching `CachingDeleteLoader`.
- **Impact:** C++ cannot benefit from delete file caching across multiple data files in the same task. Every data file triggers a fresh load of the same position/equality delete files. Acknowledged as TODO in `delete_filter.h:214`.
- **Fix:** Implement the TODO. Add a constructor overload or setter accepting a `std::function<std::unique_ptr<DeleteLoader>()>` factory.

### Incomplete Features

#### file_scan_task_reader.h:43 — Missing "\_deleted" column mode
- **Description:** `FileScanTaskReader` always filters deleted rows. Java supports a mode that emits all rows with an `_deleted` boolean column. Acknowledged by TODO at line 43.
- **Java reference:** `DeleteFilter.java` — `hasIsDeletedColumn`, `markRowDeleted()`, `columnIsDeletedPosition()`. Java's `Deletes.markDeleted()` produces the marked-delete iterable.
- **Priority:** medium

#### file_scan_task_reader.h:44 — Missing residual expression filter application
- **Description:** The `FileScanTaskReader` does not apply residual expression filters from the scan task. Acknowledged by TODO at line 44. The `FileScanTask` provides `residual_filter()` but it's only forwarded to the data reader reader options — no evaluator-based row-level filtering is done.
- **Java reference:** Java scan task readers use `ResidualEvaluator` from `org.apache.iceberg.expressions.ResidualEvaluator` to apply remaining filter predicates after file-level filtering.
- **Priority:** medium

#### delete_loader.cc:173-175 — Deletion vector support not implemented
- **Description:** `LoadDV()` is a stub returning `NotSupported`. Full DV support is needed for V3 spec compatibility.
- **Java reference:** `BaseDeleteLoader.readDV()` — reads raw bytes from the file at the content offset/length, deserializes via `PositionDeleteIndex.deserialize()`.
- **Priority:** high

#### equality_delete_writer.h:53 — Missing encryption key metadata
- **Description:** `EqualityDeleteWriterOptions` has a TODO comment: `// TODO(anyone): add key_metadata for encryption`. Neither `DataWriterOptions` nor `EqualityDeleteWriterOptions` include an encryption key metadata field.
- **Java reference:** Java's `PositionDeleteWriter` constructor takes `EncryptionKeyMetadata keyMetadata` and includes it in the built `DeleteFile`. Similarly, `DataWriter` equivalents in Java include encryption support.
- **Priority:** medium

#### position_delete_writer.cc:70 — Missing sort order validation
- **Description:** `WriteDelete()` at line 70 has a TODO: `// TODO(anyone): check if the sort order of file_path and pos observes the spec.` No validation that incoming position deletes are ordered by (file_path, position) as required by the Iceberg V2 spec.
- **Java reference:** Java's `SortingPositionOnlyDeleteWriter` explicitly handles ordering; `PositionDeleteWriter` documents that "all incoming records are ordered by file and position as required by the spec" but trusts the caller.
- **Priority:** low

#### writer.h — No single-row or iterable-row write convenience methods
- **Description:** C++ `FileWriter` only supports `Write(ArrowArray* data)` for batch writes. Java's `FileWriter<T, R>` provides both `write(T row)` for single rows and a default `write(Iterable<T> rows)` that iterates. There is no equivalent convenience API in C++.
- **Java reference:** `FileWriter.java` — `default void write(Iterable<T> rows)` and `void write(T row)`.
- **Priority:** low

#### delete_filter.h:214-217 — No injectable DeleteLoader factory (caching hook)
- **Description:** The C++ `DeleteFilter` has a TODO for exposing a `std::function<DeleteLoader()>` factory so callers can inject caching delete loaders. Java supports this via overridable `newDeleteLoader()`.
- **Java reference:** `DeleteFilter.java:161-163` — protected `newDeleteLoader()` method.
- **Priority:** medium

## Recommendations

1. **Fix the reader resource leak** in `delete_loader.cc` by wrapping the Reader in an RAII guard or using `defer`-style cleanup to ensure `Close()` is called on all paths.

2. **Implement deletion vector (DV) support** in `delete_loader.cc` — this is a high-priority gap that makes C++ incompatible with tables using V3 DVs. The `PositionDeleteIndex` infrastructure exists; only the raw-byte reading and deserialization is missing.

3. **Implement delete file caching** in `DeleteLoader` — expose the factory hook TODO and implement a caching variant. Without caching, merge-on-read performance will be significantly worse than Java for workloads where a single position delete file covers multiple data files.

4. **Add `_deleted` column mode** to `DeleteFilter` and `FileScanTaskReader` — this aligns with Java's flexible filter/mark-deleted dual mode and enables engine integrations that need to see all rows.

5. **Fix the metrics filtering** in `position_delete_writer.cc` by also erasing `column_sizes` for the metadata columns (file_path and pos). This aligns with Java's `copyWithoutFieldCounts` behavior.

6. **Implement residual expression filter application** in `FileScanTaskReader` — leverage the existing expression evaluator infrastructure to apply scan-level predicates after delete filtering.

7. **Add encryption key metadata support** to `DataWriterOptions` and `EqualityDeleteWriterOptions` so encrypted writes are possible.

8. **Replace `ICEBERG_PRECHECK` with status-returning checks** in `ComputeAliveRows` and `WriteDelete` for recoverable error conditions that should not crash the process.
