# Daily Analysis Report: Arrow

## Findings

### Logic Errors

#### metadata_column_util.cc:49 — Integer overflow in row position computation
- **Severity:** low
- **Description:** `MakeRowPositionArray` computes `start_position + i` in a loop where both operands are `int64_t`. If `start_position` is near `INT64_MAX` and `num_rows` is large, signed integer overflow produces undefined behavior. While practical Iceberg tables won't approach this limit, it is technically UB in C++.
- **Fix:** Add a bounds check before the loop: `if (start_position > std::numeric_limits<int64_t>::max() - num_rows) return InvalidArgument(...);`, or use `__builtin_add_overflow` for a checked add per iteration.

#### arrow_io.cc:175-195 — TOCTOU race condition in ReadAt (buffer variant)
- **Severity:** low
- **Description:** `InputStreamAdapter::ReadAt(int64_t position, int64_t nbytes)` (the `shared_ptr<Buffer>` variant) releases the mutex after the first `CheckOpenLocked()` call, then performs buffer allocation outside the lock, and re-acquires the lock for the actual read. If the adapter is closed between the two lock acquisitions, the buffer allocation is wasted (but the read will fail with a proper error since `CheckOpenLocked` is called again). The void-pointer `ReadAt` variant holds the lock throughout, which is the correct pattern.
- **Fix:** Either hold the lock throughout (allocate before locking, or use a shared lock), or accept the wasted-allocation path as documented behavior. Since this doesn't produce wrong results, priority is low.

### Java Inconsistencies

#### arrow_io.cc / arrow_register.cc vs ResolvingFileIO.java — Missing scheme-based FileIO routing
- **Severity:** medium
- **C++ behavior:** `FileIORegistry` provides name-based registration/lookup (`"arrow-fs-local"`, `"arrow-fs-s3"`). FileIO is resolved by explicit name; there is no scheme-to-impl mapping (e.g., `s3://` → S3FileIO).
- **Java behavior:** `ResolvingFileIO` automatically routes `s3://`, `s3a://`, `gs://`, `abfs://` URIs to the correct delegate FileIO implementation. It also has a fallback to `HadoopFileIO` when the scheme-specific implementation fails to load.
- **Impact:** C++ users must know and specify the exact FileIO name; there is no scheme-based automatic routing. This is a usability gap compared to Java.
- **Fix:** Add a `ResolvingFileIO` or scheme-to-impl mapping layer in C++. At minimum, add a helper that derives the FileIO name from a URI.

#### Arrow module lacks Iceberg-to-Arrow schema conversion vs ArrowSchemaUtil.java
- **Severity:** medium
- **C++ behavior:** The C++ Arrow module (`src/iceberg/arrow/`) does not contain schema conversion between Iceberg types and Arrow types. Schema conversion likely lives in the avro or parquet modules, or is not yet implemented.
- **Java behavior:** `ArrowSchemaUtil.convert(Schema)` converts an Iceberg schema to an Arrow Schema, handling all primitive types (INTEGER, LONG, FLOAT, DOUBLE, DECIMAL, STRING, BINARY, FIXED, BOOLEAN, TIME, TIMESTAMP, TIMESTAMP_NANO, DATE, UUID) and nested types (STRUCT, LIST, MAP). MAP type includes `"originalType": "mapType"` metadata to distinguish it from List<Struct>.
- **Impact:** Users of the C++ Arrow bundle may not have a clear path for Iceberg-to-Arrow schema conversion at the Arrow module level.
- **Fix:** Add an `ArrowSchemaUtil` equivalent in the C++ Arrow module, or document clearly where schema conversion lives.

#### S3 property key naming convention vs Java S3FileIO
- **Severity:** low
- **C++ behavior:** `S3Properties` defines keys like `s3.access-key-id`, `s3.secret-access-key`, `s3.session-token`, `s3.region`, `s3.endpoint`, `s3.path-style-access`, `s3.ssl.enabled`, `s3.connect-timeout-ms`, `s3.socket-timeout-ms`.
- **Java behavior:** Java's AWS module uses a different property naming convention aligned with the Hadoop S3A filesystem or AWS SDK properties.
- **Impact:** When migrating configuration between Java and C++ Iceberg clients, S3 property names differ. However, the C++ names follow the Iceberg spec, so this is actually correct — Java's naming is the legacy one.
- **Fix:** Document the property key mapping for users migrating from Java to C++.

### Incomplete Features

#### arrow_s3_file_io.cc — No IAM role / instance profile / credential chain support
- **Description:** The S3 FileIO only handles explicit access key + secret key authentication. It does not support AWS IAM roles, EC2 instance profiles, or credential provider chains. Arrow's `S3FileSystem` does support automatic credential resolution, but this code path always requires explicit keys.
- **Java reference:** Java's `S3FileIO` supports credential chains, IAM roles, instance profiles, and Web Identity Token credentials via the AWS SDK's default credential provider chain.
- **Priority:** medium

#### arrow_io.cc / arrow_io_util.h — No GCS or ADLS cloud storage FileIO
- **Description:** C++ only provides Local and S3 FileIO implementations. No Google Cloud Storage (GCS) or Azure Data Lake Storage (ADLS) support.
- **Java reference:** Java's `ResolvingFileIO` routes `gs://`, `abfs://`, `abfss://`, `wasb://`, `wasbs://` URIs to dedicated FileIO implementations.
- **Priority:** low (S3 is primary use case; GCS/ADLS are secondary)

#### metadata_column_util.cc — File path array uses full string copies per row
- **Description:** `MakeFilePathArray` creates a StringArray with N identical copies of the file path string. For large batches, this is memory-inefficient. A dictionary-encoded array or a zero-copy constant array approach would be more efficient. Arrow's `RunEndEncoded` type or `DictionaryArray` could be used.
- **Java reference:** Java's vectorized readers may use more efficient representations. The Java `ArrowVectorAccessor` and `VectorHolder` patterns optimize data access.
- **Priority:** low (optimization, not a correctness issue)

#### arrow_c_data_util.cc / DictEncodedArrowConverter.java — No dict-encoded vector conversion
- **Description:** When dictionary-encoded Arrow vectors need conversion to typed Arrow vectors (e.g., dict-encoded INT64 → BigIntVector), Java provides `DictEncodedArrowConverter` that handles DECIMAL, TIMESTAMP, TIMESTAMP_NANO, LONG, FLOAT, DOUBLE, STRING, BINARY, and TIME type conversions. C++ has no equivalent in this module.
- **Java reference:** `DictEncodedArrowConverter.toArrowVector()` handles 9 type categories with full null bitmap handling.
- **Priority:** low (this is a different architectural layer — C++ may not need this at this module level)

## Recommendations

1. **Add overflow checks** to `MakeRowPositionArray` in `metadata_column_util.cc` for the `start_position + i` loop to avoid theoretical integer overflow.
2. **Consider holding the mutex** throughout `InputStreamAdapter::ReadAt(buffer)` to avoid the TOCTOU pattern, or document the intentional double-lock design.
3. **Implement scheme-based FileIO routing** (like Java's `ResolvingFileIO`) to improve usability — at minimum, map `s3://` and `file://` URI schemes to registered FileIO implementations.
4. **Document the Arrow-to-Iceberg schema conversion path** — if schema conversion lives in another module (e.g., parquet), note this clearly in the Arrow module documentation.
5. **Consider adding IAM/credential chain support** for S3 in a future iteration, since this is a significant gap for AWS production deployments.
6. **Use dictionary encoding** for the `_file` metadata column array to reduce memory overhead for large batches.
