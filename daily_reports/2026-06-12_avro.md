# Daily Analysis Report: Avro

## Findings

### Logic Errors

#### avro_schema_util_internal.h:103 — HasNoIds() returns wrong value
- **Severity:** high
- **Description:** `HasIdVisitor::HasNoIds()` returns `total_fields_ == 0`, but `total_fields_` counts all fields encountered (regardless of whether they have IDs). It should check `fields_with_id_ == 0` to correctly determine if no fields have IDs. With the current bug, `HasNoIds()` returns `false` for any schema containing records/arrays/maps — even if none of their fields carry field IDs. This causes the reader path in `avro_reader.cc:245-257` to fall through to the `AllHaveIds()` check, which also returns false, and the reader errors out with "Not all fields in the Avro file schema have field IDs" instead of applying name mapping.
- **Fix:** Change `HasNoIds()` to return `fields_with_id_ == 0`:
  ```cpp
  bool HasNoIds() const { return fields_with_id_ == 0; }
  ```

#### avro_stream_internal.cc:50-53 — Read errors silently swallowed as EOF
- **Severity:** medium
- **Description:** In `AvroInputStream::next()`, when `input_stream_->Read()` fails (non-OK status or reads 0 bytes), the function returns `false`. Avro treats `false` as EOF, so a read error is indistinguishable from end-of-file. The TODO comment at line 50-52 acknowledges this limitation. If a transient I/O error occurs mid-file, the reader will silently treat it as EOF and return a truncated result.
- **Fix:** Add error tracking (e.g., an `error_occurred_` flag) and expose it via a dedicated error-check method so callers can distinguish EOF from I/O failure. Alternatively, log the error and consider adding a last-error accessor to the stream class.

### Java Inconsistencies

#### avro_reader.cc:321-336 vs AvroIterable.java:65-74 — Metadata access semantics differ
- **Severity:** medium
- **C++ behavior:** `AvroReader::Metadata()` reads metadata from the currently-open Avro reader backend. If the backend is closed, it returns an error "Reader is not opened". It can be called at any point while the reader is open.
- **Java behavior:** `AvroIterable.getMetadata()` lazily opens a **separate** `DataFileReader` to read metadata, then closes it. This is independent of the main iteration reader and can be called before or after iteration.
- **Impact:** C++ callers cannot call Metadata() after Close() (e.g., to get metadata after reading all data). Java callers can. The C++ approach is more restrictive.
- **Fix:** Either cache metadata during Open() and return it regardless of reader state, or document that Metadata() must be called before Close().

#### avro_writer.cc:311 vs AvroFileAppender.java — split_offsets() returns empty
- **Severity:** low
- **C++ behavior:** `AvroWriter::split_offsets()` returns an empty vector `{}`.
- **Java behavior:** `AvroFileAppender` does not have a `split_offsets()` equivalent at the appender level (Java handles splits at a higher level in `OutputFileFactory`). The `FileAppender` interface in Java doesn't expose split offsets.
- **Impact:** The C++ Writer interface declares `split_offsets()` but the Avro implementation returns empty — this could cause callers expecting non-empty offsets to silently produce incorrect results when splitting Avro files. If Avro doesn't support split offsets, the method should return an error or the interface should make it optional.
- **Fix:** Either implement split offset collection during writes (track sync marker positions), or document that Avro doesn't support split offsets and return a clear status.

#### avro_schema_util.cc:603-604 — Decimal precision comparison direction differs from Java
- **Severity:** medium
- **C++ behavior:** In `ValidateAvroSchemaEvolution()` line 603-604:
  ```cpp
  if (decimal_type.scale() == logical_type.scale() &&
      decimal_type.precision() >= logical_type.precision()) {
  ```
  This requires the expected type's precision to be **>=** the file's precision (promotion allowed).
- **Java behavior:** Java Iceberg's `SchemaToType` class validates decimal schema evolution. The precision check allows expected precision >= file precision (same direction as C++), so this is actually consistent. However, the scale must match exactly in both.
- **Impact:** This is actually consistent with Java. However, the condition only allows precision promotion — if the expected precision is lower than the file precision, it fails even though a rounding/conversion might be possible. This is consistent with Java's conservative approach.

### Incomplete Features

#### avro_metrics.cc:24-28 — Metrics computation is a stub
- **Description:** `AvroMetrics::GetMetrics()` returns only `row_count` and ignores the schema and metrics_config parameters. The TODO at line 26-27 notes this will be populated later if the datum writer is a MetricsAwareDatumWriter.
- **Java reference:** `AvroMetrics.fromWriter()` at `AvroMetrics.java:35-37` is also a stub with the same TODO, returning only `row_count`. Both implementations are equally incomplete.
- **Priority:** medium

#### avro_data_util.cc:563, 571, 649 — LARGE_STRING and LARGE_BINARY Arrow types not supported
- **Description:** Three TODO markers indicate that Arrow's `LARGE_STRING`, `LARGE_BINARY`, and `LARGE_LIST` types are not supported in the `ExtractDatumFromArray` function (GenericDatum write path). The direct encoder path also lacks support for these types.
- **Java reference:** Java's ValueWriters handles `CharSequence` (which includes both String and Utf8) and `ByteBuffer` (which handles both binary and large binary). Java's approach is more flexible.
- **Priority:** medium (for large data use cases)

#### avro_direct_encoder.cc:366 — AVRO_ENUM encoding not implemented
- **Description:** The `EncodeArrowToAvro` function explicitly returns `NotSupported("ENUM type encoding not yet implemented")` for AVRO_ENUM nodes. This means Avro schemas with enum fields cannot be written.
- **Java reference:** Java's `ValueWriters` supports enums via a dedicated enum writer. While Iceberg schemas don't natively use enums, Avro schemas from external sources may contain them.
- **Priority:** low (enums are rarely used in Iceberg contexts)

#### avro_writer.h:311 — split_offsets() stub
- **Description:** `AvroWriter::split_offsets()` returns an empty vector. The Writer interface declares this method but Avro doesn't populate it, so callers relying on split offset information for Avro files won't get meaningful data.
- **Java reference:** Java doesn't expose split offsets at the FileAppender level — splits are handled at a higher abstraction.
- **Priority:** low

#### avro_direct_encoder.h — EncodeArrowToAvro takes Type& but signature uses Schema& in writer
- **Description:** The `EncodeArrowToAvro` function takes `const Type&` but in `avro_writer.cc:116`, it's called with `*write_schema_` which is an `::iceberg::Schema` (which inherits from StructType, which inherits from Type). The function recurses on child types. However, there's no validation that the Iceberg type matches the Avro node type at the top level — this validation only happens deeper in the recursion. An initial type/node mismatch could produce confusing errors.
- **Priority:** low

#### avro_reader.cc:214 — TODO for basic reader metrics
- **Description:** A TODO comment at line 214 (`// TODO(gang.wu): collect basic reader metrics`) indicates that reader-side metrics collection is planned but not yet implemented.
- **Java reference:** Java's `AvroIterable` doesn't collect reader metrics either — metrics are collected at the writer level.
- **Priority:** low

#### avro_schema_util.cc:637 — Workaround for Windows linking issue
- **Description:** A comment at line 637: `// XXX: Result<::avro::NodePtr> leads to unresolved external symbol error on Windows.` This forced `UnwrapUnion` to use an output parameter pattern instead of returning `Result<::avro::NodePtr>`. This is a platform-specific workaround that deviates from the project's preferred `Result<T>` return pattern.
- **Priority:** low (workaround, not a missing feature)

## Recommendations
1. **Fix `HasNoIds()` immediately** — this is a high-severity bug that silently corrupts the read path for Avro files without field IDs, preventing name mapping from being applied correctly.
2. **Add error tracking to `AvroInputStream`** — at minimum, log read errors so that silent truncation can be detected. Consider adding a `has_error()` accessor so callers can distinguish EOF from I/O failure.
3. **Consider caching metadata during Open()** — this would align C++ `Metadata()` behavior with Java's approach and allow metadata access after Close().
4. **Prioritize `split_offsets()` implementation or documentation** — if Avro doesn't support split offsets, document this clearly and consider having the method return an error rather than an empty vector.
5. **Implement LARGE_STRING/LARGE_BINARY support** — these are common Arrow types that will be needed for production use with large datasets.
6. **Implement metrics collection** — though Java is also a stub here, having proper column-level metrics is important for query optimization. The `MetricsAwareDatumWriter` pattern from Java should be followed.
