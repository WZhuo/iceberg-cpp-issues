# Daily Analysis Report: Catalog

## Summary

## Findings

### Logic Errors

#### error_handlers.cc:55-57 — 405/406 status codes fall through to generic RestError
- **Severity:** medium
- **Description:** In `DefaultErrorHandler::Accept()`, HTTP status codes 405 (Method Not Allowed) and 406 (Not Acceptable) hit a `break` statement and fall through to the generic `RestError` path instead of being mapped to specific error kinds. Java maps 405 to `NotAllowedException` and 406 to `UnsupportedOperationException`. This means REST API clients get a generic error for unsupported methods instead of a specific, actionable error.
- **Fix:** Map 405 to a `NotAllowed` or `NotSupported` error kind, and 406 to an appropriate error kind.

#### rest_catalog.cc:236, 330 — Unreachable return statements after pagination loops
- **Severity:** low
- **Description:** Both `ListNamespaces()` (line 236) and `ListTables()` (line 330) have unreachable `return result;` statements after their respective `while(true)` loops. The loops always return from within via either `return result;` when the page token is empty or an error path. The dead code doesn't cause bugs but indicates incomplete loop structure. Either the loop should be restructured to `while (!next_token.empty())` or the unreachable return should be removed.
- **Fix:** Remove the unreachable `return result;` statements, or restructure the loops to exit naturally.

#### in_memory_catalog.cc:248 — DropNamespace error message misleading
- **Severity:** low
- **Description:** When `DropNamespace()` finds the namespace is not empty, the error message says `"{} has other sub-namespaces and cannot be deleted"`. This message is misleading because it only mentions sub-namespaces, but the check also fails if the namespace contains tables. The message should say "sub-namespaces or tables" to match the actual condition.
- **Fix:** Change error message to `"Namespace {} is not empty and cannot be deleted"`.

#### json_serde.cc:315 — delete-file-references index validation uses signed ref
- **Severity:** low
- **Description:** In `FileScanTasksFromJson()`, the indexing check `if (ref < 0 || static_cast<size_t>(ref) >= delete_files.size())` checks `ref < 0` first. While this correctly catches negative indices, the `int32_t ref` variable from `GetJsonValue<std::vector<int32_t>>` is physically an index that shouldn't be negative in well-formed data. However, if a malicious or malformed server response provides negative indices, the code correctly rejects them but the descriptive error message doesn't indicate what the expected range is.
- **Fix:** Add expected range to the error message.

#### auth_session.cc:273 — active_refresh_count_ uninitialized
- **Severity:** low
- **Description:** `OAuth2AuthSession` has member `int active_refresh_count_ = 0;` (line 277) which is correctly initialized. However, the `token_` string (line 266) is default-constructed (empty), which is fine. The `expires_at_` is initialized to epoch (line 268), which means `CalculateRefreshDelay()` will treat an unset token as already expired. This is fine in the current code path since `SetInitialToken` is always called before `ScheduleRefresh`, but could be fragile if someone calls `ScheduleRefresh` without setting a token first. No actual bug today but worth noting.
- **Fix:** Consider adding an `ICEBERG_DCHECK` in `ScheduleRefresh()` that `!token_.empty()`.

### Java Inconsistencies

#### catalog.h:166 vs Catalog.java:307 — DropTable return type differs
- **Severity:** medium
- **C++ behavior:** `DropTable()` returns `Status` — success indicates the operation completed, but doesn't distinguish between "table was dropped" and "table didn't exist."
- **Java behavior:** `dropTable()` returns `boolean` — `true` if the table was dropped, `false` if the table did not exist.
- **Impact:** Callers cannot determine whether the table actually existed and was removed. The `InMemoryCatalog::UnregisterTable` silently returns `Status{}` even if the table wasn't found (line 311 erases without checking), meaning `DropTable` reports "success" for non-existent tables.
- **Fix:** Either change return type to `Result<bool>` to match Java semantics, or at minimum ensure all catalog implementations return an error for dropping a non-existent table.

#### catalog.h:175 vs Catalog.java:317 — RenameTable return type differs
- **Severity:** medium
- **C++ behavior:** `RenameTable()` returns `Status`.
- **Java behavior:** `renameTable()` returns `void` and throws exceptions (`NoSuchTableException`, `AlreadyExistsException`).
- **Impact:** The C++ return type is fine (errors map to status), but the `InMemoryCatalog` implementation returns `NotImplemented("rename table")` (line 521) — in Java, renaming is a fully supported operation. This is more of an incompleteness issue for the in-memory catalog.
- **Fix:** Implement `RenameTable` in `InMemoryCatalog`.

#### catalog.h vs SupportsNamespaces.java — Namespace property API is split differently
- **Severity:** low
- **C++ behavior:** Namespace property updates use a single `UpdateNamespaceProperties()` method with updates map and removals set.
- **Java behavior:** Namespace property updates use two separate methods: `setProperties(Namespace, Map)` and `removeProperties(Namespace, Set)`.
- **Impact:** The C++ approach is more efficient (single round-trip) and matches the REST API spec, while Java's approach is more granular. Not a correctness issue, but an API design difference worth documenting.
- **Fix:** No fix needed; this is an intentional design choice matching the REST spec.

#### catalog.h vs Catalog.java — Missing default implementations
- **Severity:** low
- **C++ behavior:** All methods in the `Catalog` base class are pure virtual, forcing every catalog implementation to provide every method.
- **Java behavior:** Many methods have default implementations: `tableExists()` falls back to `loadTable()`, `registerTable()` throws `UnsupportedOperationException` by default, `createTable()` has overloads without sort order, `initialize()` is a no-op, `invalidateTable()` is a no-op.
- **Impact:** C++ catalog implementers must implement every method manually. The `InMemoryCatalog::RenameTable` returns `NotImplemented` — in Java this could benefit from a default throwing implementation.
- **Fix:** Consider adding default implementations for `TableExists()` (fall back to `LoadTable`) and `RegisterTable()` (throw unsupported) in the base class.

#### catalog.h vs Catalog.java:339 — Missing `invalidateTable`
- **Severity:** low
- **C++ behavior:** No `InvalidateTable` or equivalent method exists in the C++ Catalog interface.
- **Java behavior:** `default void invalidateTable(TableIdentifier identifier) {}` allows dropping cached metadata for a table.
- **Impact:** C++ catalogs cannot support cache invalidation for previously loaded tables. This matters for long-running processes where table metadata may change.
- **Fix:** Add `virtual Status InvalidateTable(const TableIdentifier& identifier)` to the Catalog base class.

#### catalog.h vs Catalog.java:190-449 — Missing TableBuilder pattern
- **Severity:** low
- **C++ behavior:** C++ has `StageCreateTable()` for transactions but no `TableBuilder` pattern for fluent table creation with `createOrReplace`, `replace`, etc.
- **Java behavior:** Java provides a rich `TableBuilder` interface with `create()`, `createTransaction()`, `replaceTransaction()`, `createOrReplaceTransaction()` methods, and multiple overloads of `newCreateTableTransaction`/`newReplaceTableTransaction`.
- **Impact:** C++ lacks the "create or replace" and "replace" transaction semantics. Users must use `UpdateTable` with create requirements instead, which is less ergonomic.
- **Fix:** Consider adding builder-pattern methods in a future enhancement.

### Incomplete Features

#### in_memory_catalog.cc:521 — RenameTable not implemented
- **Description:** `InMemoryCatalog::RenameTable` returns `NotImplemented("rename table")`. The method signature exists but no logic to move a table from one identifier to another.
- **Java reference:** `Catalog.renameTable()` is fully supported in Java's equivalent `InMemoryCatalog`.
- **Priority:** medium

#### in_memory_catalog.cc:513-514 — DropTable ignores purge flag
- **Description:** When `purge` is true, the function has a `TODO` comment but does not delete metadata files. It only unregisters the table from the in-memory store. Java's implementation either deletes data and metadata files when purge is true.
- **Java reference:** Java's `dropTable(identifier, true)` deletes all data and metadata files.
- **Priority:** medium

#### sql_catalog.cc:511-512 — DropTable ignores purge flag
- **Description:** Similar to InMemoryCatalog, `SqlCatalog::DropTable` has a TODO comment for purge support but does not delete table data/metadata files.
- **Priority:** medium

#### hive/iceberg_hive_export.h — Hive catalog not implemented
- **Description:** Only the export header file exists; there is no implementation for a Hive Metastore-backed catalog. This is a major gap compared to Java which has full Hive catalog support.
- **Java reference:** Java has a full `HiveCatalog` implementation supporting all catalog operations via the Hive Metastore.
- **Priority:** high

#### auth_manager.cc:155-156 — OAuth2 Token Exchange not implemented
- **Description:** Two TODO comments in `OAuth2Manager` indicate that `TableSession()` and `ContextualSession()` overrides are not yet implemented for token exchange (RFC 8693). This means per-table and per-context auth sessions currently return the parent catalog session as-is, preventing fine-grained access control.
- **Java reference:** Java REST catalog supports token exchange for table-scoped credentials.
- **Priority:** medium

#### auth_managers.cc — SigV4 auth type registered but not implemented
- **Description:** `KnownAuthTypes()` includes `kAuthTypeSigV4` but `CreateDefaultRegistry()` does not register a factory for it. This causes `Load()` to return `NotImplemented("Authentication type 'sigv4' is not yet supported")`.
- **Priority:** medium

#### types.h:188 — StorageCredential support missing
- **Description:** `LoadTableResult` has a TODO for `std::shared_ptr<StorageCredential>` to support table-scoped storage credentials returned by the server. All three catalogs (Rest, InMemory, Sql) ignore per-table credential configuration.
- **Priority:** medium

#### catalog_store_sqlpp23_internal.h — No config.h include path resolved
- **Description:** The sql_catalog.cc file has `#include "iceberg/catalog/sql/config.h"` (line 26) but no such file was found among the module files. This compilation dependency is either resolved by a build-time generated config header or indicates a missing file. Worth verifying.
- **Priority:** low

## Recommendations
1. **Implement `RenameTable` in `InMemoryCatalog`** — it's a core catalog operation that is currently stubbed out. The implementation is straightforward: move the metadata location entry from old identifier to new identifier.
2. **Implement purge support in `DropTable`** for both `InMemoryCatalog` and `SqlCatalog` — this is needed for proper data lifecycle management and to match Java behavior.
3. **Complete the Hive catalog implementation** — this is a major catalog type used in many deployments and its absence limits adoption.
4. **Add `InvalidateTable` to the Catalog interface** — needed for cache coherence in long-running processes.
5. **Fix the `DefaultErrorHandler::Accept`** 405/406 fall-through to provide more specific error responses.
6. **Implement SigV4 auth manager** — needed for AWS environments where IAM-based authentication is required.
7. **Consider adding default implementations** to the C++ Catalog base class for methods like `TableExists` (fall back to LoadTable) to reduce boilerplate in catalog implementations.
8. **Review `DropTable`/`DropNamespace` return semantics** — consider whether these should return `Result<bool>` to indicate whether the resource actually existed, matching Java's behavior.
